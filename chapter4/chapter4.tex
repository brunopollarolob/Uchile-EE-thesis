\chapter{F-engine design}
\label{chap:fengine}

The development of the F-engine firmware for this project utilized the Xilinx System Generator for DSP \citep{Xilinx_UG948_2020}. This process combined MATLAB Simulink and Xilinx blocks with components from the CASPER\footnote{\url{https://casper.berkeley.edu/wiki/Main_Page}} framework, enabling an efficient and modular design approach. CASPER offers a library of pre-built modules for common DSP operations, such as FFTs, PFBs, and filtering, that can be easily combined to implement complex signal processing pipelines for radio astronomy applications. An important advantage of this approach is that CASPER blocks have been extensively verified and validated through years of deployment in operational radio astronomy instruments worldwide \citep{hickish2016decadedevelopingradioastronomyinstrumentation,Parsons_2008}. This established track record means that the core signal processing functionality does not require re-verification from first principles, allowing development efforts to focus on system integration and application-specific requirements.

This approach allows digital architectures to be designed using functional blocks and visual interconnections, greatly simplifying development compared to traditional \glspl{hdl} design. The System Generator toolchain provides integrated simulation and compilation capabilities that automatically translate Simulink designs into synthesizable \gls{hdl} code for FPGA implementation. When functionalities not available in the standard CASPER or Xilinx block libraries were required, custom \gls{hdl} modules written in Verilog were developed (listed in Annex~\ref{appendix:verilog_codes}) and integrated into the Simulink model using System Generator's \textit{Black Box} feature, which allows external \gls{hdl} code to be incorporated seamlessly into the overall design flow.

\section{F-engine architecture}
\label{sec:f_engine_architecture}
The F-engine architecture was carefully structured to support the overall FX correlator approach adopted in the CHARTS digital backend (see \S\ref{sec:charts_digital_backend}) and meet the system requirements declared in \S\ref{sec:objectives}. In this architecture, the F-engine is responsible for the initial channelization of the digitized antenna signals multiplexed by the \gls{fdm} boards.

A general block diagram of the F-engine architecture is shown in Figure~\ref{fig:f_engine_block_diagram}. The design processes 32 antenna signals using the four RFSoC ADCs, each sampled at \SI{4.9152}{\giga\hertz} in quadrature I/Q mode (see \S\ref{sec:quadrature_sampling}). The digitized data is first channelized using a biplex \gls{pfb} with 4 taps, producing 8192 frequency channels per stream. These outputs are then fed into a 4-stream FFT block, which computes the \gls{fft} for each stream in parallel. The resulting spectra are re-quantized to a compact $4+4$ bit format before being transmitted via a QSFP28 optical interface. Additionally, the design incorporates custom blocks for spectral reordering, which facilitates data formatting to ensure compatibility with the X-engine and \texttt{kotekan}.

The total resource usage of this model on the RFSoC 4x2 is summarized in Table~\ref{tab:f_engine_resource_usage}, showing that the design fits comfortably within the available resources of the device, leaving room for future expansions or additional functionalities.

\section{Digitization}
\label{sec:digitization}
The digitization strategy for CHARTS was carefully designed to meet the system requirements while ensuring architectural compatibility. To achieve this, the RFSoC 4x2's \gls{rfdc} was configured in quadrature I/Q mode. This approach was chosen because real sampling would have required a rate exceeding twice the target bandwidth ($\SI{2366.4}{\mega\hertz}$), which, although within the ADC's maximum capability of 5 GSPS, would not allow a power-of-two number of parallel samples, something that is crucial for the posterior PFB/FFT. Quadrature sampling with decimation provided an efficient solution, as it enables the effective processed bandwidth to match the sampling frequency, fully capturing the required band while adhering to the RFDC's architectural constraints.

The sampling rate was therefore fixed at $\nu_s=\SI{4.9152}{\giga\hertz}$, comfortably covering the $\sim\SI{2366}{\mega\hertz}$ input band produced by the FDM stage. An internal NCO at \SI{-1.2288}{\giga\hertz} ($-\nu_s/4$) downconverts the signal to baseband, producing complex I/Q outputs centered at 0 Hz. To reduce the data rate and allow a power-of-two number of parallel samples, a decimation factor of 2 was applied, resulting in a usable baseband bandwidth of \SI{2457.6}{\mega\hertz}. This configuration is illustrated in Figure~\ref{fig:quadrature_sampling_rfdc}, which shows how the input spectrum is progressively shifted and decimated into the final baseband representation.

Since the system must simultaneously digitize 32 antennas, with 8 signals multiplexed per ADC, all converters available in the RFSoC 4x2 are used. In particular, the dual tiles 224 and 226 were enabled and configured for quadrature sampling (see Figure \ref{fig:rfsoc_tiles} for tile organization of ADCs). Table \ref{tab:rfdc_config} summarizes the configuration parameters for each ADC.

With these configurations, the output data rate is \SI{256}{bits} per clock cycle, corresponding to 8 complex I/Q samples of \SI{32}{bits} each (16 + 16 signed), giving a required AXI4-Stream clock frequency of \SI{307.2}{\mega\hertz} to sustain the \SI{2457.6}{\mega\hertz} bandwidth. In order to produce this sample clock in the ADC tiles of the RFSoC, the Texas Instruments LMX2594 RF synthesizer \citep{LMX2594} was configured to generate a \SI{307.2}{\mega\hertz} reference, used by the ADC tile. This frequency is different from the \SI{491.52}{\mega\hertz} default reference clock provided in every CASPER image instance (\texttt{rfsoc4x2rfsoc4x2\_LMX\_REF\_245M76\_OUT\_491M52.txt} file), but due to harmonics of this clock appearing after channelization, it was changed to \SI{307.2}{\mega\hertz} so that the spurious tone falls exactly on an FFT bin, simplifying its removal in later stages.

Since for dual-tile platforms (such as the RFSoC 4x2) in I/Q digital output modes the in-phase and quadrature data are produced from different ports, before channelization the data stream is reordered using the CASPER \texttt{munge} block. This ensures the stream is formatted as interleaved complex pairs starting from the oldest sample $[I_0, Q_0, I_1, Q_1, \dots, I_7, Q_7]$, compatible with the specific input requirements of the downstream FFT blocks. The real ($I$) and imaginary ($Q$) parts of the 8 parallel samples coming from the output of the RFDC are first combined with a \texttt{bus\_create} block. The resulting concatenated input bus is organized with all in-phase samples first (from newest $I_7$ to oldest $I_0$), followed by all quadrature samples (from newest $Q_7$ to oldest $Q_0$). To achieve the desired temporal ordering (oldest samples in MSB), the \texttt{munge} block is configured with the division packing order \texttt{[7, 15, 6, 14, 5, 13, 4, 12, 3, 11, 2, 10, 1, 9, 0, 8]}. As detailed in Table~\ref{tab:munge_reorder}, this configuration picks the specific slices from the input bus to construct an output where the time index increases from MSB to LSB.

\section{LO generation}

The \glspl{lo} are a key element in the \gls{fdm} architecture, as they enable the upconversion of the \qtyrange{300}{500}{\mega\hertz} signals from the antennas into higher frequency bands ranging from 300 to $\sim 2366$ MHz. Since the RFSoC 4x2 incorporates high-speed \glspl{dac}, these were used to synthesize the \glspl{lo} digitally and output them directly in the analog domain. This choice avoids the need for external frequency synthesizers, while providing high flexibility and real-time configurability.

The system requires four independent \glspl{lo} at \SIlist{1065.6;1332;1600.8;1864.8}{\mega\hertz}, in phase. The rationale behind these values will be discussed later in the channelization section (\S\ref{sec:channelization}), but they were chosen to satisfy the bandwidth and frequency placement requirements of the instrument.

The \gls{dac} tiles were configured similarly to the \gls{adc} tiles, but operating in real-to-real mode without interpolation. Each tile outputs 16 parallel 16-bit samples per clock cycle, with a sampling rate of \SI{4915.2}{\mega\hertz}, therefore the AXI4-Stream clock frequency (\SI{307.2}{\mega\hertz}, FPGA clock frequency) is also set to \SI{307.2}{\mega\hertz} to match the ADC sample clock. The DAC output is thus a real-valued waveform with a Nyquist frequency of \SI{2457.6}{\mega\hertz}, which is sufficient to cover the highest required \gls{lo} frequency of \SI{1864.8}{\mega\hertz}. The configuration parameters of the DAC tiles are summarized in Table~\ref{tab:dac_config}.

The generation of the \glspl{lo} is based on the digital synthesis of four sinusoidal tones per \gls{dac}. Each tone has programmable frequency and amplitude, and all are summed digitally to form a composite signal. At the analog stage, external bandpass filters are used to isolate the desired frequency component, ensuring that each mixer receives its corresponding \gls{lo}.

The tones are synthesized using \glspl{lut} stored in \gls{bram}, which contain the sampled values of a sine function. These are addressed by phase accumulators operating at the system clock, where the phase increment determines the output frequency. Amplitude control is achieved by applying digital attenuations to the waveform samples before summation. Both frequency and attenuation parameters are mapped to software-controlled registers, enabling their adjustment in real-time. Credits to Frederik Brecht from University of Toronto for the original tone generator design, later optimized and adapted by the author.

Figure~\ref{fig:lo_generation} illustrates the architecture of the \gls{lo} generation block. Both DAC tiles contain a sum of four tones. The outputs are connected to the appropriate pins on the RFSoC board (DAC A and DAC B), which are routed to SMA connectors for external filtering and distribution to the FDM boards.

\section{Channelization}
\label{sec:channelization}

Channelization is performed by a 4-tap PFB (see \S\ref{sec:pfbs}) with a Hamming window (see Table \ref{tab:window_functions}), convergent rounding and 18-bit coefficients. The output format of the PFB is set to 18 + 18 bits (real and imaginary parts), following CASPER design recommendations to balance precision and resource usage. 

The filtered samples are then passed to an \num{8192}-point radix-2 pipeline FFT, which operates directly on the complex input. The FFT is configured with automatic bit growth: at each stage of the 13-stage radix-2 pipeline, the word length of every sample increases by 1 bit to accommodate intermediate dynamic range expansion. Consequently, the final output has a format of $31 + 31$ bits, which is subsequently re-quantized according to the strategy described in \S\ref{sec:re-quantization}. 

The FFT length of $N=8192$ is determined by the number of pipeline stages: since $N=2^{13}$, a 13-stage radix-2 architecture is employed. The resulting channel width is computed from the effective bandwidth and the number of channels:
\begin{equation}
	\Delta\nu_\text{ch} = \frac{\Delta\nu_\text{FFT}}{N} = \frac{\SI{2457.6}{\mega\hertz}}{8192} = \SI{300}{\kilo\hertz},
\end{equation}
where $\Delta\nu_\text{FFT}=\SI{2457.6}{\mega\hertz}$ is the usable baseband bandwidth after decimation (see \S\ref{sec:digitization}). The FFT is clocked at \SI{307.2}{\mega\hertz} with 8 samples per clock cycle, producing a new spectrum every:
\begin{equation}
	t_s=\frac{N}{\nu_\text{clk} \times N_\text{parallel}}=\frac{8192}{\SI{307.2}{\mega\hertz}\times8}=3.\overline{3}\,\unit{\micro\second},
\end{equation}
where the overline denotes a repeating decimal. Unlike real-input FFT architectures, no channels are discarded as the complex I/Q input naturally produces a full-spectrum output (see \S\ref{sec:dft}). Key specifications of the F-engine design are summarized in Table~\ref{tab:f_engine_metrics}.

For a maximum expected \gls{dm} of $\sim\SI{800}{\parsec \per \centi \meter \cubed}$ at the lowest frequency of \SI{300}{\mega\hertz} and after upchannelization to $\sim16\,\text{k}$ channels ($\Delta \nu_\text{ch} \approx \SI{12.3}{\kilo\hertz}$), the intra-channel dispersion smearing using Eq.\ \ref{eq:intra_channel_smearing} is approximately \SI{3}{\milli\second}, which is acceptable for the target science cases.

\section{Re-quantization and calibration}
\label{sec:re-quantization}

The FFT output produces complex samples with a precision of $31+31$ bits per channel, which must be reduced before transmission to the X-engine. To achieve this, the system applies a re-quantization step that truncates the data stream to $4+4$ bits. Although this quantization inevitably leads to some information loss, in practice the effect is small for typical radio astronomy signals and is compensated by the advantages of reducing the data volume. The lower bit depth enables higher bandwidth coverage and efficient real-time processing, while keeping quantization-induced biases within negligible levels.

Nevertheless, this introduces systematic effects in both amplitude and phase of the measured correlations, which in turn impact calibration accuracy and the fidelity of the visibilities. To minimize these effects, the system applies a digital gain and scaling stage prior to truncation. The calibration aims to adjust each frequency channel such that the signal level is close to 1-bit \gls{rms}. In this regime, the correlation between the signal and the quantization error becomes vanishingly small, the increase in output variance is only about 4\%, and the resulting bias in correlation measurements falls below $10^{-6}$ in magnitude and $10^{-11}$ degrees in phase (see \citealt{menaparra2018quantizationbiasdigitalcorrelators} for a detailed analysis of quantization effects in radio interferometry).

The calibration process operates in three steps, as illustrated in Figure~\ref{fig:quantizer_architecture}. First, the $31+31$ bit FFT output of each channel is multiplied by a 16-bit programmable gain factor, applied identically to the real and imaginary components. This stage allows fine adjustment of the signal amplitude per channel. The result is a $48+48$ bit word\footnote{The additional bit comes from the signed extension of the gain value to 17 bits.}, which is then shifted left by a configurable number of bits (equivalent to multiplying by powers of two). This left shift is applied uniformly across all frequency channels and serves to optimize the dynamic range before truncation. Finally, the system selects the 4 most significant bits of the real and imaginary components and applies convergent rounding-to-even following the UG901 guidelines \citep{Xilinx_UG901_2025} for hardware efficiency coding, yielding a compact $4+4$ bit representation per channel. A clipping flag is raised whenever a value with integer part equal to \texttt{0111} (decimal 7) and a non-zero fractional component reaches the rounding stage\footnote{The possible range of values for a 4 bit signed number is $[-8,7]$, hence the need for clipping.}, providing diagnostics for potential saturation.

The combination of the programmable per-channel gain and the global left shift defines the effective digital gain applied in the calibration stage. These parameters are software-configurable, allowing flexible tuning of the re-quantization pipeline. The calibration gains are initially computed by measuring the RMS power of the channelized data for each input and frequency channel, ensuring that the levels are brought close to the desired 1 bit RMS target. After the initial power equalization, a digital calibration loop refines the per-channel gain coefficients to ensure that the amplitude of each frequency bin approaches the target level of 1~bit~RMS within a predefined tolerance. This process is iterative and operates directly on the FPGA through a closed feedback scheme. subsection~\ref{sec:digital_gain_calibration} details the calibration procedure.

\subsection{Digital gains calibration procedure}
\label{sec:digital_gain_calibration}
At each calibration iteration $m$, the system measures the average RMS magnitude of the channelized spectra across $N_{\mathrm{acc}}$ short integrations:
\begin{equation}
    \text{RMS}_k^{(m)} = \sqrt{\frac{1}{N_{\mathrm{acc}}} \sum_{n=0}^{N_{\mathrm{acc}}-1} \lVert X_{kn} \rVert^2},
\end{equation}
where $X_{kn}$ denotes the complex FFT output of frequency channel $k$ at frame $n$. The integration length $N_{\mathrm{acc}}$ is chosen to ensure sufficient SNR and stability, typically set to \num{300000} frames (\SI{1}{\second} of data).

From a control theory perspective, the calibration algorithm is implemented as a \emph{discrete-time integral (I) controller} operating on the gain magnitude \citep[see][Section~3.3]{franklin1998digital}. Unlike static proportional correction, an integral action is required to maintain a non-zero gain value when the error converges to zero. The deviation of the measured $\text{RMS}_k^{(m)}$ from the target $\text{RMS}_\mathrm{t}$ drives the accumulation of the control signal.

To prevent steady-state limit cycles caused by the coarse integer quantization of the FPGA registers (actuator quantization), the system employs a \textit{software-based high-precision integrator}. The gain coefficients are maintained in software as floating-point variables $\tilde{g}_k$, updated via a multiplicative law:
\begin{equation}
    \tilde{g}_k^{(m+1)} = \tilde{g}_k^{(m)} \cdot \left[ 1 + K \left( \frac{\text{RMS}_\mathrm{t}}{\text{RMS}_k^{(m)}} - 1 \right) \right].
\end{equation}

Here, $K$ represents the integral gain parameter (set to $0.5$). This value was tuned to provide an overdamped response, effectively acting as a low-pass filter that smooths out the stochastic variance inherent in the RMS estimation of RFI-like signals. The physical integer gain $g_k$ loaded into the FPGA is derived by quantizing the internal integrator state:
\begin{equation}
    g_k^{(m+1)} = \text{clip}\left( \text{round}\left( \tilde{g}_k^{(m+1)} \right), 0, G_{\mathrm{max}} \right).
\end{equation}

As described in \citet{franklin1998digital}, Chapter~10, separating the internal high-precision state from the quantized actuator allows the controller to accumulate fractional corrections over multiple cycles. This effectively eliminates the deadband effect where small errors would otherwise fail to trigger a bit-flip in the integer registers. Figure~\ref{fig:calibration_algorithm} illustrates the hybrid control loop architecture, combining software-based high-precision integration with hardware-applied quantized gains.

	\includegraphics[width=\textwidth]{gains_cal_fig/cal_algorithm.pdf}
	\caption[Block diagram of the hybrid gain control loop.]{Block diagram of the hybrid gain control loop. The software domain maintains a floating-point state $\tilde{g}_k$ acting as a virtual integrator to accumulate fractional corrections, while the hardware domain applies the quantized integer gain $g_k$. The loop gain $K$ dampens the stochastic measurement noise.}
	\label{fig:calibration_algorithm}

In practice, the calibration operates only over active frequency bands defined by a binary mask, excluding out-of-chain channels to avoid noise bias. By using a proportional control law, the step size dynamically scales with the error magnitude, taking larger steps when far from the target and finer steps near convergence. This approach significantly reduces the number of iterations required to reach the desired calibration state compared to fixed-step methods.

A global convergence condition is defined by the fraction of active frequency channels satisfying:
\begin{equation}
    \left| \text{RMS}_k^{(m)} - \text{RMS}_\mathrm{t} \right| \le \epsilon, 
\end{equation}
where $\epsilon$ is a small tolerance threshold (typically set to $0.1$ units RMS). When more than $99\%$ of the active channels meet this criterion, the calibration is considered complete. Once the iterative loop converges, the final per-channel gain vector:
\begin{equation}
\mathbf{g} = 
\begin{bmatrix}
g_0 \\
g_1 \\
\vdots \\
g_{N-1}
\end{bmatrix},
\end{equation}
is stored in a \texttt{.npy} file for subsequent use during system initialization.

However, the raw per-channel gains obtained from the iterative calibration process can display unphysical fluctuations, often caused by measurement noise, RFI, or other systematic effects. To mitigate this, a smoothing step is applied to the gain vector $\mathbf{g}$ to enforce spectral continuity and suppress spurious variations. This is achieved by applying an iterative algorithm that combines polynomial fitting with Fourier-based filtering, as described in \citet{Hsyu2016}. The result is a set of per-channel gains that are smooth across frequency, effectively suppressing channel-to-channel noise while preserving genuine spectral features. It is important to clarify that this smoothing stage has not been implemented fully in the current firmware version, but is planned for future releases. Nevertheless, the existing calibration procedure already achieves satisfactory performance in practice.

This digital gain calibration needs to be performed only once for each input and frequency channel under stable observing conditions. Updates may be required if instrumental responses drift significantly over time or if the RFI environment changes.

\section{Data reorder}
Following the FFT and re-quantization stages, the data streams from all 32 antennas must be synchronized, aggregated, and formatted for transmission. This process involves buffering to align the FDM chains, correcting spectral inversions, and performing a logical transposition (corner turn) to organize the data by frequency channel.

\subsection{Data format and throughput}
The data streaming implementation selects a specific subset of the full \num{8192}-channel spectrum to optimize the output data rate. Rather than transmitting the full bandwidth, the system selects \num{672} channels per antenna (corresponding to the FDM chains defined in Table~\ref{tab:fdm_chains}) from each $3.\overline{3}\,\unit{\micro\second}$ spectrum frame. Channels outside the science bands, such as the \qtyrange{0}{300}{\mega\hertz} region and guard bands between FDM chains, are discarded. With 4 ADCs per board and 32 antennas in total, this results in \num{21504} transmitted channels per frame. 

At the hardware level, the system operates at a clock frequency of \SI{307.2}{\mega\hertz}, with each FFT core outputting 8 parallel samples per clock cycle. After the re-quantization stage described in \S\ref{sec:re-quantization}, each sample is represented as a complex number in $4+4$ bit format, resulting in 8 bits per sample. This configuration yields a localized data width of 64 bits per cycle per ADC ($8 \text{ samples} \times 8 \text{ bits/sample}$). With four ADCs operating simultaneously, the theoretical peak aggregate throughput reaches $4\times\SI{19.66}{\giga\bit\per\second}=\SI{78.64}{\giga\bit\per\second}$.

In practice, the actual sustained data rate is lower because valid samples are produced only during bursts corresponding to the active FDM chains, rather than continuously across all \num{8192} FFT bins. The system transmits \num{21504} active channels (\num{672} channels per antenna $\times$ 32 antennas), each carrying 8 bits of data, every $3.\overline{3}\,\unit{\micro\second}$. This results in an average sustained data rate of approximately \SI{51.6}{\giga\bit\per\second} for the entire board (see Eq.\ \ref{eq:board_data_rate}).

\subsection{FFT shift}
The FFT core outputs frequency bins sequentially over 1024 cycles per frame, following the standard FFT convention where bins are numbered \qtyrange{0}{8191}{}. Due to the quadrature sampling employed, bin 0 represents DC (\SI{0}{\mega\hertz}) and bin 8191 represents \SI{2457.3}{\mega\hertz}. However, the internal FFT implementation outputs these bins in a wrapped order: first the upper half of the spectrum (bins \qtyrange{4096}{8191}{}, corresponding to \qtyrange{1228.8}{2457.3}{\mega\hertz}) followed by the lower half (bins \qtyrange{0}{4095}{}, corresponding to \qtyrange{0}{1228.5}{\mega\hertz}). This ordering creates a problem for contiguous transmission of FDM chains: certain chains span the boundary between these two segments, resulting in their data being split across non-adjacent portions of the output stream. Most notably, FDM chain 3, which covers \qtyrange{1099.2}{1300.8}{\mega\hertz}, has its upper portion (bins \qtyrange{4096}{4336}{}) appearing at the beginning of the upper frequency segment, while its lower portion (bins \qtyrange{3664}{4095}{}) appears at the end of the lower frequency segment. To ensure that all FDM chains can be transmitted as contiguous blocks without fragmentation, a dedicated FFT reordering module is implemented in hardware to rearrange the output spectrum into ascending frequency order (bins \qtyrange{0}{8191}{}).

The reordering module employs a double-buffering scheme using two \glspl{bram} to guarantee zero data loss while performing the frequency-domain shift. While one buffer is written with incoming FFT data in the wrapped order (bins \qtyrange{4096}{8191}{} followed by bins \qtyrange{0}{4095}{}), the other is read out in ascending frequency order (bins \qtyrange{0}{8191}{}). This operation effectively performs an FFT shift that ensures all FDM chains are transmitted as contiguous blocks without fragmentation. A sync toggle signal controls the switching between write and read modes for each buffer, ensuring continuous operation without stalls or data loss. A critical function of this stage is the correction of spectral inversion. The analog mixing scheme (see Table~\ref{tab:fdm_lo_tones}) results in specific FDM chains having a flipped and conjugated spectrum. To standardize the data:
\begin{itemize}
    \item \textbf{Direct chains (0, 4--7):} Preserve spectral orientation and are written into standard \gls{fifo} buffers.
    \item \textbf{Inverted chains (1--3):} Are written into \gls{lifo} buffers. Reading from a LIFO effectively time-reverses the sample sequence, restoring the frequency bins to their natural ascending order. A conjugation operation is simultaneously applied to correct the phase inversion.
\end{itemize}

Once all FDM chains have been written, the buffers are read out in parallel. Each buffer provides a 64-bit word (8 complex samples) per clock cycle. With 32 antennas, this yields a massive aggregate bus width of \num{2048} bits per cycle.

\subsection{Data structure definitions}

To formalize the organization of this aggregate data, we define the following notation:
\begin{itemize}
    \item $k \in \{0,\dots,671\}$: Absolute frequency channel index.
    \item $(\nu_a)_k$: The complex data sample for antenna $a$ at frequency channel $k$, represented in 4+4 bit format.
\end{itemize}

We define the spatial vector $\antvec{k}$ as the simultaneous set of samples from all 32 antennas for a single frequency channel $k$:
\begin{equation}
    \antvec{k} = \begin{bmatrix}
        (\nu_0)_k \\
        (\nu_1)_k \\
        \vdots \\
        (\nu_{31})_k
    \end{bmatrix}.
\end{equation}
This vector is 256 bits wide ($32 \times 8$ bits) and represents the fundamental unit of data for downstream correlation, as it contains all spatial information for a single frequency bin.

\subsection{First stage corner turn}
\label{sec:corner_turn}
The raw output from the memory buffers is organized by antenna, but downstream processing requires data organized by frequency. Specifically, at any given read cycle $m$, the 32 buffers output a 2048-bit matrix $\mathbf{K}_m \in \mathbb{C}^{32 \times 8}$ where rows correspond to antennas and columns to the 8 parallel frequency channels:
\begin{equation}
	\mathbf{K}_m = \begin{bmatrix}
		(\nu_0)_{8m} & (\nu_0)_{8m+1} & \cdots & (\nu_0)_{8m+7} \\
		(\nu_1)_{8m} & (\nu_1)_{8m+1} & \cdots & (\nu_1)_{8m+7} \\
		\vdots & \vdots & \ddots & \vdots \\
		(\nu_{31})_{8m} & (\nu_{31})_{8m+1} & \cdots & (\nu_{31})_{8m+7}
	\end{bmatrix}_{32 \times 8}.
\end{equation}
To group samples by frequency (i.e., to form spatial vectors), the hardware performs a logical transposition (corner turn). This operation swaps the dimensions of the data matrix:
\begin{equation}
	\mathbf{K}_m^\top = \begin{bmatrix}
		(\nu_0)_{8m} & (\nu_1)_{8m} & \cdots & (\nu_{31})_{8m} \\
		(\nu_0)_{8m+1} & (\nu_1)_{8m+1} & \cdots & (\nu_{31})_{8m+1} \\
		\vdots & \vdots & \ddots & \vdots \\
		(\nu_0)_{8m+7} & (\nu_1)_{8m+7} & \cdots & (\nu_{31})_{8m+7}
	\end{bmatrix}_{8 \times 32}.
	\label{eq:corner_turn}
\end{equation}
Each row of the transposed matrix $\mathbf{K}_m^\top$ now corresponds exactly to a spatial vector $\antvec{8m+j}^\top$ for $j=0,\dots,7$. This transposition ensures that the packet payload is serialized as a sequence of complete frequency channels.

To implement this corner turn efficiently in hardware, we utilized the CASPER \texttt{munge} block, which permits arbitrary bit-level reordering of the data bus (similar to the reordering done in Table \ref{tab:munge_reorder}). For this specific application, the block is configured to treat the 2048-bit input bus as 256 distinct 8-bit slices (corresponding to the $32 \times 8$ complex samples). The division packing order required to perform the matrix transposition is defined by the permutation vector generated with the following MATLAB command:
\begin{lstlisting}[style=Matlab-editor, basicstyle=\small\ttfamily, caption={MATLAB code to generate the permutation vector for the corner turn operation.}]
	reshape(reshape(0:255, 8, [])', 1, [])
\end{lstlisting}


This configuration effectively rewires the bus connections in the FPGA fabric, ensuring that the data flowing into the packetizer is correctly formatted as contiguous spatial vectors.

\section{Packetizer architecture}

The streaming system is implemented using a high-throughput packetization block that performs data-width conversion, deep buffering, and protocol header insertion compatible with the 100\,GbE interface. The architecture is designed to handle the bursty nature of the aggregated spectral data, using a programmable inter-packet delay to ensure stable operation. The dataflow proceeds through:

\begin{enumerate}

	\item \textbf{Width conversion and buffering}.  
	The aggregated spectral data arrives as a massive 2048-bit wide word (containing 8 frequency channels for all 32 antennas). A \gls{piso} converter buffers these words into a deep internal FIFO (configured with 8192 entries) and serializes them into 512-bit words required by the 100\,GbE core. This large buffer depth provides the necessary elasticity to absorb the high-bandwidth bursts from the FFT reordering stage while the output is throttled by the Ethernet link rate.

	\item \textbf{Packetizer \gls{fsm}}.  
	The core logic is governed by a three-state \gls{fsm} that orchestrates packet formation and timing:

	\begin{itemize}
		\item \textbf{WAIT (Inter-packet gap)}.  
		The \gls{fsm} rests in this state between packets. It utilizes a counter to enforce a programmable delay defined by \texttt{sleep\_cycles}. This idle window is crucial to prevent buffer overflows in the downstream \gls{mac} hardware, effectively acting as a rate-limiting mechanism. Once the counter reaches the target delay, the \gls{fsm} transitions to HEADER.

		\item \textbf{HEADER (Metadata injection)}.  
		Before transmitting the spectral payload, the \gls{fsm} synthesizes a 512-bit header word. This header contains the current FPGA timestamp (seconds and microseconds), the sub-band ID (\texttt{band\_idx}, $\ell$), and a 64-bit sequence number (\texttt{seq\_num}, $n$) for packet loss tracking. This word is injected directly into the output stream, ensuring precise metadata association for every packet.

		\item \textbf{READ (Payload transmission)}.  
		The \gls{fsm} reads 512-bit words from the PISO buffer and asserts \texttt{tx\_valid}. Transmission continues until the number of words sent equals the configured \texttt{pkt\_len}. Upon completion, the \gls{fsm} asserts the end-of-frame (\texttt{tx\_eof}) signal, increments the sub-band and sequence counters, and returns to the WAIT state.
	\end{itemize}

\end{enumerate}

A critical design choice in this architecture is the reliance on the \texttt{sleep\_cycles} parameter. Practical experience with CASPER 100\,GbE cores suggests that maintaining a consistent, tunable inter-packet gap is often more effective at preventing internal MAC buffer overflows than reactive flow control, which can introduce latency jitter. By tuning \texttt{sleep\_cycles}, the system creates a ``FLUSH'' equivalent period that allows the 100\,GbE core to process the previous frame fully before new data arrives. \subsection{Packet segmentation and layout}

To adhere to \gls{mtu} constraints of Ethernet packets (typically limited to $\sim\SI{9}{\kilo\byte}$) and facilitate downstream processing across four independent nodes (see Figure~\ref{fig:charts_signal_chain}), the full \num{21504}-channel spectrum is segmented into four sub-bands ($\ell=0,\dots,3$) of \num{5376} channels. Each packet thus carries exactly one quarter of the total bandwidth, allowing each processing node to handle one sub-band independently. Every sub-band is transmitted as a separate UDP packet with a payload of 85 512-bit words (\texttt{pkt\_len}=85). Table~\ref{tab:subbands} details this frequency mapping.

\section{Timing and synchronization}
\label{sec:timing_sync}

As detailed in Table~\ref{tab:ethernet_packet_structure}, the metadata header of every transmitted packet includes two dedicated fields for this purpose: a 64-bit \textit{timestamp} (split into seconds and microseconds) and a 64-bit \textit{sequence number}. The F-engine employs these fields to reconstruct the precise UTC time of every frame.

The system's synchronization strategy is designed in two phases: an initial software-based implementation for laboratory validation, and a high-precision hardware-based mode for on-sky deployment, which leverages an external \gls{gps}-disciplined clock and will be detailed in \S~\ref{sec:gps_sync}.

\subsection{Current implementation: software-triggered synchronization}

In the current validation phase, the system operates without an external time reference, such as a \gls{gps} disciplined clock. Therefore, synchronization is established through a software trigger initiated by the control computer.

During initialization, a Python control script captures the current UNIX epoch time, $T_0$, using the system clock. This value is written to the FPGA's control registers mapped to the header timestamp fields. Crucially, the write operation triggers a synchronous reset of the internal frame counter or sequence number ($n$). From this instant onwards, the \texttt{timestamp\_sec} and \texttt{timestamp\_usec} fields in the packet header remain static, containing $T_0$, while the \texttt{seq\_num} field increments for every processed spectrum frame.

Since each spectrum is formed by 4 packets, the sequence number $n$ must be divided by 4 and rounded to the nearest integer to determine the corresponding spectrum frame index. The exact time $t_{\text{packet}}$ of any received packet is reconstructed offline using the known frame duration $t_{s}=3.\overline{3}\,\si{\micro\second}$:
\begin{equation}
	t_{\text{packet}} =  T_0 + \left\lfloor \frac{n}{4} \right\rfloor \cdot 3.\overline{3}\,\si{\micro\second}
\end{equation}

While this method is sufficient for verifying coarse timestamping and short-term stability, its absolute timing accuracy is limited by the non-deterministic latency of the software-to-hardware write transaction (typically in the millisecond range), which introduces a constant but unknown offset from true UTC.
