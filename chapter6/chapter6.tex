\chapter{Conclusions and future work}
\label{ch:conclusions}
This work successfully detailed the design and implementation of the F-engine for the \gls{charts} project, fulfilling all major objectives defined for the digital backend’s first stage.

The implementation integrates several innovations over the state of the art, including the use of a complex FFT for I/Q sampling and the demultiplexing of multiple antennas with a single ADC via the \gls{fdm} scheme. Regarding the specifics objectives declared at the beginning of this thesis, here are the achievements of each of them:

\begin{itemize}
    \item The implementation of a quadrature digitization scheme operating over a bandwidth of 2457.6 MHz, encompassing the full band delivered by the FDM stage.
    \item The digital generation of the four local oscillators required by the \gls{fdm} mixers using a single RFSoC DAC port, avoiding the need for external frequency synthesizers and providing real-time configurability. The LO generation module achieved SFDR of 39.31 dB, enough to operate the mixers in the FDM without significant leaks.
    \item The output spectra were successfully requantized to a compact $4+4$ bit format after applying digital gains calibration. This enabled efficient data throughput while minimizing quantization bias.
    \item A comprehensive data aggregation and formatting pipeline was developed, which aligns FDM chains, corrects spectral inversions, performs a corner turn, and discards unused channels between FDM chains. The CPS phases of the noise source test over the optical fiber confirmed that the alignment procedures were effective.
    \item Development of a high-throughput packetizer that enables data transmission via a QSFP28 optical interface at $\sim\qty{52.63}{\giga\bit\per\second}$, transmitting data streams with a $\sim0.004\%$ packet loss.
\end{itemize}

Validation tests confirmed the integrity and coherence of the system. Laboratory tests, including the 8-antenna correlation test, validated the phase stability and effectiveness of the digital gain calibration procedure necessary for interferometric observations. Furthermore, the successful solar transit observation confirmed the expected interferometric fringe rates and demonstrated phase stability and predicted rotations across the full digital chain in an on-sky environment. The recovered projected baseline matched the nominal separation of 20 meters to within a few centimeters.

The results of this work validate the architectural approach of CHARTS and its F-engine component, providing a flexible, modular, and scalable foundation for the array to achieve its scientific objectives.

\section{Future work}

While the current F-engine meets its primary performance requirements, several developments are necessary to scale the system toward a full CHARTS deployment. The most critical areas involve board synchronization, timing infrastructure, and multi-board data routing for the X-engine.

\subsection{Synchronization across multiple RFSoC boards}
\label{sec:gps_sync}
Scaling beyond a single board requires tight timing synchronization between several RFSoC so that all ADCs sample with a common time and frequency reference. The planned approach is to use a GPS disciplined timing system, such as the TCG 01-G\footnote{\url{https://www.microchip.com/en-us/product/tcg_01-g}} or the SyncServer S650\footnote{\url{https://www.microchip.com/en-us/products/clock-and-timing/systems/gnss-timing-instruments/syncserver-s650}}, which provides:

\begin{itemize}
    \item a 10 MHz frequency reference,
    \item a 1 PPS timing pulse, and
    \item an \gls{irigb} stream.
\end{itemize}

The 10~MHz signal would discipline all RFSoC clocks, ensuring identical ADC sampling rates across boards. Achieving this requires modifying the configuration of the LMK04828 clock synthesizer: the device must be reprogrammed (via TI’s TICS PRO\footnote{\url{https://www.ti.com/tool/TICSPRO-SW}}) to use CLKIN0 as the primary reference input, replacing the current internal SI5395B clock routing through CLKIN1 (see Figure~\ref{fig:rfsoc_clocking_scheme}).

For the operational phase, the timing logic will be upgraded to support hard-synchronization via the external GPS clock. The FPGA design will include an input for the \gls{pps} signal, which will serve as the heartbeat of the system.

In this mode, the internal counters are decoupled from software control and driven directly by the hardware clock:
\begin{itemize}
    \item \textbf{Seconds Counter:} This counter increments strictly on the rising edge of the \gls{pps} signal, representing the integer seconds from the epoch.
    \item \textbf{Frame Counter ($n'$):} This counter resets to zero exactly at the \gls{pps} edge and increments for every spectrum frame processed within that second.
\end{itemize}

Consequently, the packet header becomes fully dynamic: the timestamp fields update every second, and the sequence number represents the intra-second frame index. The reconstruction of the packet time becomes deterministic and free from long-term drift:
\begin{equation}
    t_{\text{packet}} = \text{sec}_{\text{GPS}} + \frac{ \left\lfloor \frac{n'}{4} \right\rfloor}{\num{1200000}} \ \si{\second}
\end{equation}
where $\text{sec}_{\text{GPS}}$ is the current second count from the GPS \gls{pps} and \num{1200000} is the number of spectrum frames per second. This architecture ensures that no truncation errors accumulate over time, as the fractional time is reset every second. The absolute timing precision in this mode is limited only by the jitter of the \gls{pps} signal and the stability of the FPGA system clock, satisfying the stringent requirements for interferometric array calibration. 

To retrieve absolute time information beyond the epoch-based seconds count, the \gls{irigb} stream from the GPS timing unit will remain accessible at all times. The proposed approach involves synchronizing the FPGA counters with the absolute \gls{irigb} time each time the counters are reset (i.e., at system startup or reconfiguration). This synchronization event associates the FPGA's internal epoch with a known absolute timestamp, enabling the reconstruction of full calendar date and time for any subsequent packet. A dedicated microcontroller (e.g., an Arduino or Raspberry Pi) connected to the GPS timing unit will continuously decode the \gls{irigb} stream and make this information available to the processing nodes via a network interface, allowing backend systems to query the absolute time reference whenever needed.


\subsection{Corner turn expansion and multi-board networking}

A crucial point to emphasize is that each RFSoC board is only capable of digitizing up to 32 antennas. For the X‑engine to operate correctly, a second-stage corner turn is therefore required so that each outgoing packet contains channels from all 256 antennas. Concretely, this implies additional networking and aggregation logic: a high‑capacity switch (or dedicated backplane) must collect sub‑band streams from multiple RFSoC boards and reassemble them so that each backend node receives the full set of antennas for its assigned frequency slice. This requires packet aggregation logic that merges packets from distinct RFSoC sources into unified frames, preserves sequence numbers and timing, and handles reordering and loss recovery. The architecture and functionality would then be similar to the CHIME corner turn backplane \citep{CHIME_cornerturn_2016}, but adapted to the specific requirements and scale of CHARTS.
This multi-board networking and aggregation layer is essential for scaling the F-engine to the full CHARTS array, ensuring that the X-engine receives correctly formatted data streams for correlation and beamforming across all antennas.
